# DR-0005: FSL-1.1 license for public Metis releases

- **Status:** Accepted (Ant, 2026-06-12)
- **Date:** 2026-06-12
- **Supersedes:** the Apache-2.0 recommendation in `docs/plans/PLAN-metis-os-opensrc.md` §4
- **Superseded-by:** —

## Context

PLAN-metis-os-opensrc.md §4 recommended Apache-2.0 for the public framework release,
reasoning from enterprise adoption and the LangChain precedent. Since that plan was
written, `Busa-Legacies/metis-framework` shipped as the extracted product repo under
**FSL-1.1-Apache-2.0** (Functional Source License: source-available, free for internal
use, no competing commercial use, auto-converts to Apache-2.0 after 2 years per release).

Two licenses across the same product surface would create a seam: an Apache-2.0
public framework would let anyone commercialize exactly the pieces metis-core's
FSL protects, hollowing out the product boundary.

## Decision

**FSL-1.1-Apache-2.0 for all public Metis releases** (the #121 launch artifact and
any later phase repos), matching metis-core.

Rationale (presented 2026-06-12, Ant: "FSL for metis os too makes sense"):
- Consistent with the metis-core product seam; no license arbitrage between repos.
- Still source-available with a credible openness story (delayed-open: every release
  becomes Apache-2.0 after 2 years).
- Asymmetric reversibility: relicensing *more* permissive later is always possible;
  clawing back from Apache-2.0 is not.

## Consequences

- The launch repo ships metis-core's existing `LICENSE` file unchanged (criterion 5
  of the CC-114 go/no-go is satisfied by it).
- Community-contribution posture stays DCO-only (no CLA) per the opensrc plan.
- If enterprise adoption ever stalls on FSL, revisit with a new DR, not an in-place
  license swap.
