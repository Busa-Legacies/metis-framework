# Decision Record (DR) Standard

Last updated: 2026-06-02

## Why this exists

Git captures **what changed** (diffs + intentional commit messages). It does *not* reliably
capture **why** a cluster of changes happened: the research or decision that drove them. That
"why" lived only in a session's live context and evaporated when the session ended: the diffs
sailed into history, but the rationale was reconstructable only by archaeology, and the recall
path (working-context → memory files → RAG) returned nothing because nobody wrote it down.

**The failure that motivated this** (see [DR-0001](DR-0001-boris-cherny-config-hardening.md)):
a research session produced 5 concrete config/hook changes, all committed with good messages,
but no durable record tied the *why* to the *what*. A later session asked "what did we learn
from that?" and the natural recall path came up empty, even though the answer was sitting in
`git log` the whole time.

A Decision Record fixes this by putting the *why* into a **RAG-indexed, recall-surfaced layer**
(`docs/process/decisions/`, covered by the `shared` index → discoverable by <<MACHINE_1_ID>>, <<MACHINE_2_ID>>, and
Claude Code), linked bidirectionally to the commits that are the *what*.

## What a DR is

A short, **immutable**, numbered markdown file recording one significant decision: the context
that forced it, the decision taken, the alternatives weighed, the concrete changes it produced,
and the consequences. It is *not* a changelog entry (that's the commit) and *not* a task (that's
`task-queue.md`). It is the durable answer to **"why does the system work this way, and what
changed when we decided it?"**

## When to create one: the trigger

Create a DR when a cluster of work meets **any** of these:

1. **Research → changes.** A research/study pass (scout, a talk, an article, an investigation)
   drove a set of changes to how the system works.
2. **Doctrine/architecture change.** Work touched *how the system operates*: `CLAUDE.md`,
   `docs/process/*.md`, hooks, `settings*.json`, routing, the memory/recall architecture, the
   sync/lock model, agent protocols.
3. **A non-obvious choice between alternatives.** You picked A over B for reasons that won't be
   obvious from the diff and that a future session would otherwise have to re-derive.
4. **Ant says "this is a milestone."**

**Do NOT create a DR for:** routine task completion, bug fixes with an obvious cause, content
edits, dependency bumps, or anything whose "why" is fully self-evident from the commit message.
When unsure, ask: *"Six months from now, would someone need the reasoning, or just the diff?"*
If just the diff, no DR.

> The trigger is deliberately narrow. The #1 reason decision logs die is becoming noise: a DR
> per commit is as useless as none. Aim for a handful per month, not per session.

## Format

- **File:** `docs/process/decisions/DR-NNNN-kebab-slug.md` (zero-padded 4-digit number).
- **Numbering:** monotonic, never reused. The next number = highest existing + 1.
- **Immutable:** once `Accepted`, do not rewrite the decision. New information → a *new* DR that
  `Supersedes` the old one; the old one's status becomes `Superseded by DR-NNNN`. This preserves
  the historical reasoning instead of erasing it.
- **Fields:** see [TEMPLATE.md](TEMPLATE.md). Required: `Status`, `Date`, `Context`, `Decision`,
  `Changes`, `Consequences`. Optional: `Alternatives`, `Supersedes` / `Superseded-by`.
- **Status values:** `Proposed` · `Accepted` · `Superseded by DR-NNNN` · `Reverted`.
  - A DR that Ant approves *within the same session it is written* may start at `Accepted`; `Proposed` is only for decisions left open for later review.

## The why↔what link (bidirectional)

The DR **ID is the join key**, robust to rebases and SHA churn:

- **Commit → DR:** every commit implementing the decision carries the ID in its message, e.g.
  `feat(end-protocol): harden close from research (DR-0001)`. Find the code from the decision:
  `git log --grep=DR-0001`.
- **DR → commit:** the DR's `## Changes` section lists the touched files and the marker
  `git log --grep=DR-NNNN`. For a permanent record you may also pin explicit short SHAs (the
  helper's `link` command does this). Find the decision from the code: open the DR by ID.

## Discoverability

- **Index:** `docs/process/decisions/README.md`: one line per DR (number · status · title ·
  hook). Regenerated by the helper; this is the human entry point.
- **RAG:** the folder is under `docs/process/`, indexed as `shared`; conceptual queries
  ("why did we change the close protocol?") surface the DR automatically. Re-index after adding
  DRs (`POST /api/memory/index {"agent":"shared"}`) per the memory-surface-map.
- **Recall path:** a DR that also changes Claude-Code behavior should get a one-line pointer in
  `ClaudeCode/memory/MEMORY.md` like any other memory; the DR remains the canonical record.

## How it's created (low-friction, no separate ritual)

DR creation is folded into the **existing history-authoring steps**, never a standalone chore:

- **`/checkpoint` and `/end`** own it. When the work being committed meets the trigger above,
  the protocol creates/updates a DR *as part of* the commit, and puts the `DR-NNNN` ID in the
  commit message. (See the checkpoint/end skills and `~/.claude/CLAUDE.md`.)
- **Stop-hook nudge.** The Stop hook (`hook-alerts.sh`) runs a cheap, zero-LLM heuristic: if the
  new intentional commits touched doctrine/config paths or carry milestone keywords, it sets a
  `milestone-pending` marker. The next `/checkpoint` reminder then carries: *"this cluster looks
  milestone-shaped; consider a DR."* The human/agent still decides; the hook only surfaces it.

## Helper

`scripts/decision-record.py` (stdlib only):

```bash
scripts/decision-record.py new "Title here"           # scaffold next-numbered DR from TEMPLATE
scripts/decision-record.py new "Title" --supersedes DR-0002
scripts/decision-record.py list                        # table of all DRs + status
scripts/decision-record.py link DR-0003 [<sha>...]     # pin commit SHAs into ## Changes
scripts/decision-record.py index                       # regenerate README.md index
```

## Lifecycle summary

```
trigger met → /checkpoint or /end creates DR (Accepted) → commit carries DR-NNNN
            → helper regenerates README index → (optional) MEMORY.md pointer
new info later → new DR Supersedes old; old → "Superseded by DR-NNNN" (never deleted)
```
