## Why this exists

The old pattern ā€” `dispatch --agent smith --message "do X"` ā€” is fire-and-forget.
Good for mechanical tasks; dangerous for anything with real scope or side effects.
This protocol adds a **plan-before-dispatch** step that makes scope, model, risk,
and cost visible before anything runs.

The protocol has three modes, selected by risk tier:

| Risk | Mode | What happens |
|---|---|---|
| `low` | Immediate | Dispatch fires; no preview needed |
| `medium` | Preview-then-run | Plan shown; auto-runs after a brief window |
| `high` / `critical` | Stop-for-approval | Plan shown; explicit approval required |

## The lane plan

Before any dispatch, a **lane plan** is constructed ā€” either implicitly (logged only) or
explicitly (shown to the human). Fields:

```
Goal:         one-line description of what this lane will do
Role:         smith / scout / warden / echo / ...
Engine:       the model/tier (or "lane default")
Scope:        files/dirs in scope; "unconstrained" if not specified
Risk tier:    low / medium / high / critical
Est. cost:    rough estimate based on engine tier
Est. tokens:  rough estimate based on task length
Artifacts:    what output is expected (diff / report / test result / ...)
Approval:     auto / pending / blocked on human
```

The plan is what gets logged to the feedback corpus (`~/.openclaw/feedback/`) and what
gets linked to the task's `lane_history` on completion.

## Mission packet contract

Every dispatched lane receives a **mission packet** before the human message reaches
the model. The packet is generated by `scripts/lib/mission_packet.py` and rendered by
`scripts/dispatch`; callers should not hand-roll a competing prompt envelope.

The packet exists because local models are most reliable when the work is bounded.
It gives the lane:

- `role_contract` ā€” what this lane is for, what it may do, and what it must not do
- `task` ā€” the exact requested work
- `route` ā€” selected engine/model/runtime plus whether the output is proposal-only
- `scope` ā€” files/dirs inferred from the request, or `unconstrained`
- `risk` ā€” low/medium/high/critical with the reason
- `context` ā€” bounded snippets only; no broad memory dump
- `constraints` ā€” non-negotiable behavior rules
- `expected_output` ā€” the artifact shape the caller expects
- `verification` ā€” reminder that lane output is not task completion evidence by itself

Local Ollama lanes should produce bounded artifacts. Claude/Codex remain responsible
for judgment, applying changes to the real tree, and verification unless a future
route explicitly grants a stronger execution surface.

## Using `--preview`

Any `dispatch` call can add `--preview` to print the plan without executing:

```bash
dispatch --agent smith --engine 5.4-standard --message "refactor auth.py" --preview
```

Output:
```
[dispatch:plan]
  Goal     : refactor auth.py
  Role     : smith
  Engine   : 5.4-standard ā†’ openai/gpt-5.4 (thinking: medium)
  Risk     : medium (role default)
  Scope    : unconstrained
  Artifact : path plus fenced code block, or a clearly labeled patch proposal
  Mode     : proposal-only
  Est cost : ~$0.02
  Approval : auto-proceed
  Packet   : schema=1, route=explicit engine
[dispatch:plan] Run without --preview to execute.
```

Use `--preview` to validate routing decisions before a large or high-cost dispatch.

## Routing guidelines

Choose role and engine together ā€” they multiply. If `--engine` is omitted,
`scripts/dispatch` now resolves a default through `scripts/lib/dispatch_policy.py`.
The default is conservative:

- local Qwen (`qwen-shallow`) for bounded low-risk drafts, summaries, logs, and research
- Codex standard (`5.4-standard`) for implementation or mutation-shaped work
- Sonnet / mini Codex for review and quality checks, depending on role
- deep Codex (`5.5-deep`) for security/auth/prod or critical money/live-runtime scopes
- escalation from local to mini Codex after repeated local failures (`--local-failures 2`)

Callers can add policy hints without hardcoding a model:

```bash
dispatch --agent scout --work-type summary --message "summarize todayā€™s queue" --preview
dispatch --agent smith --work-type implementation --mutation mutates --message "add tests for scripts/dispatch" --preview
dispatch --agent warden --work-type review --risk high --message "review auth token handling" --preview
```

High and critical routes are hard-gated. `--preview` always prints the plan, but
execution requires explicit approval:

```bash
dispatch --agent warden --message "review production auth tokens" --preview
dispatch --agent warden --message "review production auth tokens" --approve-risk
```

Explicit `--engine` remains an override, but preview still shows the inferred work
type, mutation mode, risk, and route reason.

Reference table:

| Task class | Role | Engine guidance |
|---|---|---|
| Research / doc reading / pattern search | `scout` | `qwen-shallow` for bounded/local-context drafts; escalate to `5.4m-shallow` or Sonnet when source-backed accuracy matters |
| Code generation / config / boilerplate | `smith` | `qwen-shallow` only for proposal/draft output; Codex/Sonnet for actual repo edits |
| Code review / QA / security audit | `warden` | `5.4m-shallow`; `5.5-deep` for critical review |
| Task decomposition | `steward` | `5.4m-shallow` |
| Quality verdict (approve/iterate/reject) | `arbiter` | `5.4m-shallow` |
| Session logs / daily log prose | `echo` | `qwen-shallow` (local, low stakes) |
| Strategy / unclear / multi-step coordination | `main` | `5.4-standard`; `5.5-deep` for agentic |

Scale up engines only when correctness justifies cost. A scout that reads docs doesn't
need `5.5-deep`.

Current local-lane rule: Ollama output can be used as a draft, recommendation, or
review input. It cannot by itself close an implementation task. A task reaches
`done` only after a tool-bearing agent applies/verifies the work and records evidence.

## Approval gate procedure

For `high` or `critical` risk dispatches from Claude Code (not from scripts):

1. Print the lane plan (goal, role, engine, scope, risk tier, affected files)
2. State what approval is needed and why
3. Wait for explicit "yes / proceed / approve" ā€” a lack of objection is not approval
4. On approval: dispatch with the same parameters, log the approval as an artifact
5. On rejection: log the reason; propose an alternative scope reduction or risk downgrade

For `critical` (money, trading params, prod deploys): write the approval reason into
the task notes before dispatching. No verbal approval without a written trace.

## Evidence requirements

Every dispatch should link at least one artifact when it completes.
Prefer `file_diff` for code tasks, `report` for research, `test_result` for QA.
A dispatch that completes with zero artifacts is a signal that the task definition
was too vague or the result was discarded.

## Feedback Metrics

`scripts/dispatch` appends one JSONL feedback record per executed lane call. Schema
2 is additive and keeps legacy `verdict` / `correction` fields so older readers can
continue parsing old and new records. New records include:

- selected engine/model/runtime and `route_reason`
- policy metrics: `work_type`, `risk`, `mutation`, `requires_approval`, `local_failures`
- mission packet risk/route/scope/expected output
- output status, latency, artifact kind, output length
- outcome placeholder: verdict, applied/used/discarded, and escalation count

Queue-runner output artifacts also include a `Lane Metrics` JSON block. It records
selected route, arbiter verdict, final status, whether output was used/applied, and
why it was discarded when rejected or infra-failed.

Escalation is conservative: queue-runner passes accumulated local failure/reject
signals into `dispatch --local-failures`. At two local failures, low/medium local
routes escalate through the dispatch policy to mini Codex; high/critical approval
gates still apply.
